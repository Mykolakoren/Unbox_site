### 🟡 Эксплуатация

- **`deploy.sh` без аргументов первым делом запускал `deploy_back()`** — тот самый `git pull` на
  разошедшемся серверном чекауте, который запрещён основным аудитом. Теперь отказывается работать
  и печатает, как деплоить бэк правильно. `all` = только фронт. (`f3c029e`)
- **Молчаливый фолбэк на SQLite:** без `DATABASE_URL` приложение поднималось на файле рядом с кодом —
  выглядело здоровым, отдавало пустую базу, писало брони в файл, который затрёт следующий деплой.
  Теперь fail-fast в проде. (`f3c029e`)
- **`/health` врал:** возвращал константный `ok`, не проверяя БД. При OOM (458 МБ на nginx+uvicorn+
  Postgres) ядро убивает Postgres, uvicorn выживает — мониторинг и деплой-гейт горели зелёным, пока
  каждый запрос отдавал 500. Теперь `SELECT 1` и 503 при отказе. (`f3c029e`)
- **Из git убраны 4 файла `*.db`** (в `backend/database.db` лежал живой аккаунт `role=owner`) и
  `.env.local`. `.gitignore` их не защищал — они уже были tracked. (`f3c029e`)

## 2. Открыто (по убыванию риска)

1. **CRM-синк отменяет оплаченные брони без возврата денег** ([crm/sync.py:449](../backend/app/api/v1/crm/sync.py#L449)).
   Хуже: если Google вернёт пустой список (потеря доступа сервис-аккаунта), `seen_gcal_ids` будет
   пуст и снесёт **все** сессии окна разом. Нужен guard «пустой ответ → не удалять ничего».
2. **`weekly-cashback` не идемпотентен** ([telegram.py:468](../backend/app/api/v1/telegram.py#L468)) — в докстринге
   обещано поле `cashback_applied_at`, но его нет в коде. Повторный запуск начислит всем второй раз.
   `weekly_rebate` защищён выборкой, но без `UNIQUE(user_id, week_start)` — параллельный запуск пробьёт.
3. **`quick-pay` сессии CRM без защиты от двойного клика** ([crm/sessions.py:563](../backend/app/api/v1/crm/sessions.py#L563)) —
   нет `FOR UPDATE` и нет `UNIQUE(session_id)` на платеже. Двойной клик = задвоенный доход.
4. **`extend_booking` у не-админа всегда падает 500** ([routes.py:4174](../backend/app/api/v1/bookings/routes.py#L4174)) —
   `UUID(booking.user_uuid)` на поле, которое уже `UUID`. Кнопка «Продлить +30 мин» не работает вообще.
5. **`trim` оплаченной брони не обновляет `charge_amount`** → последующий `waive` возвращает полную
   исходную сумму (клиенту дарятся деньги).
6. **`consecutive_pricing` двигает баланс по `pending`-броням** — возвращает скидку за списание,
   которого ещё не было.
7. **Sentry мёртв с обеих сторон:** `sentry-sdk` нет в requirements, `VITE_SENTRY_DSN` не задан в сборке.
   Видимости ошибок в проде — ноль.
8. **`requirements.txt` в корне не содержит `slowapi`**, который импортируется в `main.py` на уровне
   модуля. При пересоздании дроплета бэкенд не стартует.
9. **Конфигов nginx и systemd нет в git.** Из репо нельзя подтвердить даже, включён ли gzip — а без него
   бандл едет по проводу несжатым. Пересоздание сервера = восстановление по памяти.
10. **Off-box бэкапа БД по-прежнему нет** (перенесено из прошлого аудита, §5#3).
11. **Ленивая загрузка карты на лендинге** — Leaflet (148 KB) сидит в критическом пути ради карты на «/».
    Вынести в `lazy()` = −148 KB, но появится плейсхолдер: это UX-решение, за владельцем.

## 4. Чек-лист деплоя этих правок

Фронт деплоится целиком и безопасно. **Бэкенд — только хирургически, по файлам.**

### Бэкенд (7 файлов + 1 новый + 2 теста)

```
backend/app/db/session.py                  # пул + fail-fast
backend/app/main.py                        # /health с SELECT 1
backend/app/api/v1/telegram.py             # sync-вебхук + пул абонемента
backend/app/api/v1/billing.py              # advisory-lock
backend/app/api/v1/bookings/routes.py      # фантомный возврат, row-lock, пул
backend/app/api/v1/users/admin.py          # топап + заморозка
backend/app/api/v1/crm/clients.py          # N+1
backend/app/db/init_data.py                # индексы
backend/app/services/billing_defer.py      # пул
backend/app/services/pricing.py            # пул
backend/app/services/subscription_pool.py  # ← НОВЫЙ ФАЙЛ
```

Порядок по §3 основного документа: для **каждого** файла сначала `scp` серверной версии и `diff`
с локальной (убедиться, что разница = только эти правки, а не снос прод-патчей), потом бэкап на
сервере → заливка → `systemctl restart` → health-check.

`subscription_pool.py` заливать **первым** — остальные файлы его импортируют, без него бэкенд не стартует.

Побочно вскрылись и починены два системных источника тихих сбоев:
- **`Booking.id` сравнивался со строкой** в трёх местах (в `pricing.py` даже с `# type: ignore`).
  Postgres молча адаптирует, любой другой драйвер падает в драйвере — из-за этого путь

## 2. Открыто (по убыванию риска)

1. **Конфигов nginx и systemd нет в git.** Из репо нельзя подтвердить даже, включён ли gzip — а без него
   бандл едет по проводу несжатым. Пересоздание сервера = восстановление по памяти.
2. **Off-box бэкапа БД по-прежнему нет** (перенесено из прошлого аудита, §5#3).
3. **`weekly_cashback` и `weekly_rebate` — одна и та же фича, реализованная дважды.** Обе кредитуют
   баланс за недельную скидку за объём; в кроне стоит только `weekly_rebate`. Сейчас они разведены
   общим журналом (дважды заплатить нельзя), но одну из двух стоит убрать. **Вопрос к владельцу.**
4. **Старые `backend/test_*.py` бьют в прод.** Не тесты, а скрипты: при импорте выполняются против
   `DATABASE_URL`, `test_pricing.py` регистрирует реальных пользователей. Переименовать в
   `scripts/check_*.py`, пока кто-нибудь не запустил `pytest` на дроплете.
5. **Ленивая загрузка карты на лендинге** — Leaflet (148 KB) в критическом пути ради карты на «/».
   Вынести в `lazy()` = −148 KB, но появится плейсхолдер: это UX-решение, за владельцем.
6. **`VITE_SENTRY_DSN` не задан в сборке фронта** — пакет есть, DSN нет, ошибки фронта не видны.
7. **Alembic-миграции** (перенесено, §5#4) — не срочно: `run_migrations()` покрывает потребность.

### ⚠️ ПЕРЕД рестартом: проверить, создадутся ли UNIQUE-индексы

`run_migrations()` создаёт два уникальных индекса. Если в проде **уже есть дубли**, индекс не создастся
(ошибка уйдёт в лог, старт не упадёт — но защиты не будет). Проверить и почистить заранее:

```sql
-- дубли платежей по одной CRM-сессии (следы двойных кликов по «Оплачено»)
SELECT session_id, count(*) FROM therapist_payments
 WHERE session_id IS NOT NULL GROUP BY session_id HAVING count(*) > 1;

-- дубли недельных начислений
SELECT user_id, week_start, count(*) FROM weekly_rebates
 GROUP BY user_id, week_start HAVING count(*) > 1;
```
Если что-то нашлось — это и есть уже случившиеся двойные списания/начисления. Разобрать вручную
(лишние строки удалить, баланс скорректировать), затем рестартовать бэкенд.

### Проверить на проде после рестарта: абонементы в camelCase

Найти пользователей, у которых пул односторонний, — это те, у кого топап был невидим для крона:
```sql
SELECT email, subscription->>'remainingHours' AS camel, subscription->>'remaining_hours' AS snake
  FROM "user"
 WHERE subscription ? 'remainingHours' AND NOT (subscription ? 'remaining_hours');
```
Новый код читает такой пул корректно и чинит при первой записи, но по этому списку стоит сверить,
не списали ли с кого-то деньги поверх оплаченного абонемента.
